### ① Part 基类

```cpp
class Part {
protected:           // protected：派生类可以访问
    int id;          // 零件编号（1~100）
    double size;     // 尺寸值（mm）
    string type;     // 类型（"P1"/"P2"/"P3"/"P4"）

public:
    Part(int i, double s, string t);  // 构造函数
    virtual ~Part() {}                 // 虚析构（确保正确释放）
    
    // getter函数（只读，不改数据）
    int getId() const { return id; }
    double getSize() const { return size; }
    string getType() const { return type; }
    
    // 虚函数：派生类可以重写，实现不同零件的不同显示格式
    virtual void print() const;
};
```

**为什么基类要写 virtual？**
- `virtual print()`：让派生类可以重写显示方式
- `virtual ~Part()`：如果用 `Part* p = new P1(...)` 然后 `delete p`，不写virtual的话只会调用Part的析构，不会调用P1的析构，可能导致内存泄漏

### ② P1~P4 派生类

```cpp
class P1 : public Part {       // public继承：Part的public成员在P1中也是public
public:
    P1(int i, double s) : Part(i, s, "P1") {}  // 调父类构造函数
    void print() const override {               // override：明确告诉编译器这是重写
        cout << "| P1 | " << id << " | " << size << " |  +  |";
    }                                           // 增环显示 +
};
// P2/P3/P4同理，只是类型字符串不同，减环显示 -
```

**P1~P4 的区别只有3个地方：**
1. 构造时传给基类的 type 不同（"P1"/"P2"/"P3"/"P4"）
2. 尺寸范围不同（写在 main 的 genP1~genP4 里）
3. print() 显示的内容不同（增环显示+，减环显示-）

### ③ Assembly 类

```cpp
class Assembly {
private:
    int id;          // 装配体编号（1~100）
    P1* p1;          // 指向这个装配体用的P1零件
    P2* p2;          // 指向P2零件
    P3* p3;          // 指向P3零件
    P4* p4;          // 指向P4零件
    double gapC0;    // 装配间隙 C0 = C1 - C2 - C3 - C4
    
    void calcGap() { gapC0 = p1->getSize() - p2->getSize() 
                              - p3->getSize() - p4->getSize(); }

public:
    Assembly(int i, P1* a, P2* b, P3* c, P4* d);
    
    // getter
    int getId() const { return id; }
    double getGap() const { return gapC0; }
    bool isOK() const { return gapC0 >= -1e-9 && gapC0 <= 0.45; }
    // 注意：用 -1e-9 而不是 0.0，因为浮点数计算可能有极其微小的误差
    
    // setter（交换零件时用）
    void setP2(P2* p) { p2 = p; calcGap(); }  // 换了P2后重新算C0
    void setP3(P3* p) { p3 = p; calcGap(); }
    void setP4(P4* p) { p4 = p; calcGap(); }
};
```

**关键点：** Assembly 里存的是**指针**（P1* p1），不是对象（P1 p1）。因为一个零件只属于一个装配体，用指针的话多个装配体共享同一个零件对象。交换零件时只需要换指针指向，不需要复制整个零件对象。

### ④ Algorithm 基类

```cpp
class Algorithm {
protected:
    vector<P1*>& p1s;  // 引用：不复制，直接操作外部的零件列表
    vector<P2*>& p2s;
    vector<P3*>& p3s;
    vector<P4*>& p4s;
    vector<Assembly*> bestSol;  // 存储最优的100个装配体
    double bestAvgGap;           // 最优方案的平均间隙
    int bestOK;                  // 最优方案的合格数

public:
    virtual void run() = 0;  // 纯虚函数：派生类必须实现
    
    void printResult(...);    // 输出结果（所有算法共用）
};
```

**为什么用引用（&）？** 因为零件列表很大（400个对象），用引用避免复制，同时让算法可以直接修改排序。

### ⑤ Greedy 类

最简单的算法，适合作为"对照组"来对比SmartMatch的效果。

```cpp
class Greedy : public Algorithm {
    void run() override {
        // 1. 全部从小到大排序
        sort(p1s, cmpAscP1);  // 按p1->getSize()升序
        sort(p2s, cmpAscP2);
        sort(p3s, cmpAscP3);
        sort(p4s, cmpAscP4);
        
        // 2. 一一配对：第i小的P1配第i小的P2/P3/P4
        for (int i = 0; i < 100; i++) {
            Assembly* a = new Assembly(i+1, p1s[i], p2s[i], p3s[i], p4s[i]);
            bestSol.push_back(a);
        }
        
        // 3. 统计
        for (每个装配体) 统计合格数和平均间隙
    }
};
```

**为什么Greedy效果差？** 小P1配小P2+P3+P4 → C0勉强合格。但是大P1配大P2+P3+P4 → P2+P3+P4的增长速度（跨度0.70）比P1（跨度0.35）快一倍，导致后面半数装配体的C0为负，不合格。

## 三、关键数学原理：C0守恒

### 为什么总C0是固定的？

每个装配体的C0 = C1 - C2 - C3 - C4

100个装配体的C0总和 = (ΣC1) - (ΣC2) - (ΣC3) - (ΣC4)

ΣC1 是所有P1的尺寸之和。不管怎么配对，100个P1全部用完，所以ΣC1不变。
同理ΣC2、ΣC3、ΣC4也不变。

**结论：总C0是常数，不随配对方式改变！**

### 这对合格率意味着什么？

设 k = 合格数，每个合格装配体的C0平均为A，不合格的平均为B：
```
k × A + (100-k) × B = 总C0 ≈ -7.4（看随机数据）
```

B的下限是-0.60（C1最小-42.85 减 C2最大10.15 减 C3最大23.15 减 C4最大10.15）。

如果A=0（合格品C0≈0），B=-0.60（不合格品顶到最负）：
```
k × 0 + (100-k) × (-0.60) = -7.4
-60 + 0.6k = -7.4
0.6k = 52.6
k = 87.7
```

**理论极限约88%！** 这就是为什么再好的算法也到不了90%以上。

## 四、SmartMatch算法（超详细伪代码）

### Phase 1：贪心搜索（构建初始解）

这一步的目标：给每个P1找到最合适的P2/P3/P4，使得C0尽可能在[0, 0.45]范围内。

**为什么P1要从大到小处理？**
- 大的P1（≈43.20mm）需要小的P2+P3+P4才能配出C0在范围内的装配体
- 小的P1（≈42.87mm）更苛刻，需要非常小的P2+P3+P4
- 所以让大的P1先选（选择面宽），小的P1后用剩下的（没办法，整体守恒限制）

```
步骤1：P1按尺寸从大到小排序
        P2/P3/P4按尺寸从小到大排序

步骤2：准备3个布尔数组 usedP2[100], usedP3[100], usedP4[100]
        全部初始化为 false（都还没用）

步骤3：for pi = 0 to 99（处理每个P1，从最大的开始）:
    |
    |-- c1 = p1s[pi]->getSize()
    |-- bestScore = -999（最小的分数，确保第一个找到的组合就会被记录）
    |-- bestJ = bestK = bestL = 0（记录最优的P2/P3/P4索引）
    |
    |-- for j = 0 to 99:        // 遍历所有P2
    |     if usedP2[j] 跳过（这个P2已经被用了）
    |     c2 = p2s[j]->getSize()
    |     |
    |     |-- for k = 0 to 99:    // 遍历所有P3
    |           if usedP3[k] 跳过
    |           c3 = p3s[k]->getSize()
    |           |
    |           |-- for l = 0 to 99:  // 遍历所有P4
    |                 if usedP4[l] 跳过
    |                 c4 = p4s[l]->getSize()
    |                 c0 = c1 - c2 - c3 - c4
    |                 
    |                 // 评分规则（重点！）
    |                 if c0 >= 0 AND c0 <= 0.45:
    |                     score = 1000 - |c0|
    |                     // 合格范围内：越接近0分越高
    |                     // c0=0.00 → 1000分
    |                     // c0=0.45 → 955分
    |                     // c0=0.225 → 977.5分
    |                 elif c0 < 0:
    |                     score = 500 + c0
    |                     // C0为负：越接近0越好
    |                     // c0=-0.01 → 499分（几乎合格！）
    |                     // c0=-0.60 → -100分（太难救了）
    |                 else (c0 > 0.45):
    |                     score = 500 - (c0 - 0.45)
    |                     // C0太大：越接近0.45越好
    |                     // c0=0.46 → 499分
    |                     // c0=1.00 → -5分
    |                 
    |                 if score > bestScore:
    |                     更新 bestScore, bestJ, bestK, bestL
    |  
    |-- 标记 usedP2[bestJ]=true, usedP3[bestK]=true, usedP4[bestL]=true
    |-- 构建装配体: new Assembly(pi+1, p1s[pi], p2s[bestJ], p3s[bestK], p4s[bestL])
    |-- 存入 bestSol
```

**为什么评分规则这么设计？**
- 合格范围内1000分起步，不合格范围内500分起步 → 算法永远优先选合格组合
- 合格范围内`1000-|c0|` → 在合格的前提下，C0越接近0越好（省"预算"给更多装配体）
- 不合格范围内`500+c0` → 实在找不到合格组合时，选最接近合格的（C0=-0.01→499分，离合格一步之遥）

### Phase 2：局部交换（从~50%提升到~84%）

经过Phase 1，大约40-60个装配体是合格的。Phase 2通过交换零件来让更多装配体合格。

```
for iter = 0 to 499:
    |
    |-- 把所有装配体分成两组：
    |    good[] = 合格的那些（C0∈[0,0.45]）
    |    bad[] = 不合格的那些
    |
    |-- if bad为空（全部合格）: 退出
    |
    |-- for 每个不合格装配体 b:
    |     for 每个合格装配体 g:
    |       |
    |       |-- 【交换P2】
    |       |   保存 b的P2 和 g的P2
    |       |   调换：b.setP2(g的P2), g.setP2(b的P2)
    |       |   重新统计合格数
    |       |   if 合格数增加: 保留交换, 进入下一轮
    |       |   else: 换回来
    |       |
    |       |-- 【交换P3】（同上逻辑）
    |       |-- 【交换P4】（同上逻辑）
    |       |-- 【同时交换P2+P3】（同时换两个，有时候单换不够）
    |       |-- 【同时交换P2+P4】
    |       |-- 【同时交换P3+P4】
    |
    |-- 如果所有bad-good组合都试过了都没改善：
    |    尝试在bad之间交换P2（可能有帮助）
    |    if 还是没改善: 退出循环（收敛了）
```

**为什么要同时交换两个零件？** 有时候只换一个P2，两个人的C0变化不够大，不够让不合格的变成合格。但同时换P2和P3，变化就大了，可能刚好让b变合格而g仍然合格。

### Phase 3：C0压缩（挤"预算"）

这一步的目标：在合格装配体之间交换零件，让它们的C0总和变小，释放出"预算"让更多不合格的变成合格。

```
for iter = 0 to 99:
    |
    |-- 在合格装配体中找两个a和b
    |-- 保存交换前的C0a和C0b
    |-- 交换a和b的P2
    |-- 算新的C0a"和C0b"
    |-- if (新C0a+新C0b) < (旧C0a+旧C0b)  // 总和变小
           AND a仍然合格 AND b仍然合格:
        保留交换（释放了预算）
    |-- else: 换回来
    
    |-- if 没有找到任何能改善的交换: 退出
```

**为什么C0总和能变小？** 交换前两个人都合格，但可能a的C0=0.15，b的C0=0.10，总和0.25。交换后a的C0=0.08，b的C0=0.07，总和0.15。释放了0.10的"预算"，可以让一个接近合格(-0.01)的装配体变得合格！

## 五、main.cpp执行流程详解

### 完整执行步骤

```
步骤1：定义零件列表
       vector<P1*> p1a;  // 存100个P1的指针
       vector<P2*> p2a;  // 存100个P2的指针
       ...

步骤2：生成零件（自动去重）
       int d1 = genUnique(p1a, 42.85, 43.20, 100);
       // 在[42.85, 43.20]范围内生成100个P1

       // 遇到重复的重新生成，d1记录重复次数
       同理生成P2/P3/P4

步骤3：显示信息
       尺寸链方程：C0 = C1 - C2 - C3 - C4

       然后释放这组零件

步骤5：重新生成零件（3次独立运行取最优）
       for att = 0 to 2:
           生成全新的一组零件
           SmartMatch sm(...)
           sm.run()
           记录合格数
           if 比之前的好: 更新最优记录
           释放这组零件

步骤6：重新生成零件（展示最终结果）
       生成全新零件
       SmartMatch sm(...)
       sm.run()
       sm.printResult()  // 显示详细配对方案

步骤7：输出对比总结
       Greedy合格数 vs SmartMatch合格数
       Greedy平均间隙 vs SmartMatch平均间隙

步骤8：cin.get()  // 等待按Enter，防止闪退
```

### 为什么3次独立运行？

每次随机生成的零件数据不同，SmartMatch的结果也不同。跑3次取最优，可以避免遇到"运气不好"的数据。

### 重复检查的机制

```cpp
template<class T>
static int genUnique(vector<T*>& v, double lo, double hi, int n) {
    set<double> seen;  // 用set记录已经生成的尺寸
    int dupAttempts = 0;
    
    while (v.size() < n) {
        double val = randIn(lo, hi);           // 随机生成一个值
        double r = round(val * 10000) / 10000; // 四舍五入到0.0001
        if (seen.find(r) != seen.end()) {       // 如果已经生成过
            dupAttempts++;                       // 计数+1
            continue;                            // 重新生成
        }
        seen.insert(r);                          // 记录这个尺寸
        v.push_back(new T(v.size()+1, val));    // 创建零件
    }
    return dupAttempts;  // 返回重复次数（显示给用户看）
}
```

**为什么用set？** set是C++标准库的"集合"，里面的元素不会重复。用 seen.find(r) 可以快速判断r是否已经在集合中。

## 六、为什么Greedy差而SmartMatch好？

### Greedy 的问题

| 装配体编号 | P1 | P2 | P3 | P4 | C0 | 合格？ |
|-----------|-----|-----|------|-----|------|-------|
| 1（最小） | 42.85 | 9.90 | 22.95 | 9.90 | 0.10 | ✅ |
| ... | ... | ... | ... | ... | ... | ... |
| 30 | 42.94 | 9.98 | 23.01 | 9.98 | -0.03 | ❌ |
| ... | ... | ... | ... | ... | ... | ... |
| 100（最大） | 43.20 | 10.15 | 23.15 | 10.15 | -0.25 | ❌ |

P2+P3+P4从42.75增长到43.45（跨度0.70），而P1只从42.85增长到43.20（跨度0.35）。P2+P3+P4涨得比P1快得多，导致后面一半的C0为负。

### SmartMatch 的优势

**核心思想：谁更需要小零件，谁先选。**

```
大P1（≈43.20）→ 搜遍所有剩余零件，找最小的P2+P3+P4 → C0≈0.15 ✅
中P1（≈43.00）→ 搜遍剩余，找合适的 → C0≈0.05 ✅
小P1（≈42.87）→ 剩下什么用什么 → C0≈-0.40 ❌
```

**效果：** 约78-84个配得好（合格），剩下的16-22个虽然不合格但已尽力。

### 理论极限

```
总C0 ≈ -7.4（由这组随机数据决定）
不合格C0最小能到 -0.60（由设计范围决定）

k × 0 + (100-k) × (-0.60) = -7.4
k ≈ 88

所以无论如何也超不过88%，84%已经很接近极限了！
```

## 七、输出结果解读

### 每一列的含义

```
| ID | C1(mm) | C2(mm) | C3(mm) | C4(mm) |  C0(mm) |Status|
|  1 | 43.197 | 10.139 | 23.045 | 10.012 |   0.0000 |  OK  |
```

- **ID**: 装配体编号（1~100）
- **C1~C4**: 这个装配体用的四个零件的尺寸
- **C0**: 装配间隙 = C1-C2-C3-C4，单位mm
- **Status**: OK=合格(C0∈[0,0.45])，BAD=不合格

### 对比总结

```
| Metric        | Greedy  | SmartMatch |
| Qualified     | 30/100  | 80/100     |  ← 合格数
| Rate          | 30%     | 80%        |  ← 成功率
| Avg Gap (mm)  | -0.0654 | -0.0660   |  ← 平均间隙（基本一样，因为守恒）
```

注意平均间隙几乎相同——这就是C0守恒的表现。SmartMatch并没有改变总C0，只是让C0的分布更合理（更多装配体落在合格范围内）。
